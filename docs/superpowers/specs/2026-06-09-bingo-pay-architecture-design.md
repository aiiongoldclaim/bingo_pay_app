# Bingo Pay вЂ” Flutter Architecture Design

**Date:** 2026-06-09
**Status:** Approved
**Platforms:** iOS + Android (mobile only)
**Flutter SDK:** ^3.10.8

---

## 1. Project Overview

Bingo Pay is a multi-vendor marketplace mobile application with two user roles:

- **Buyer** вЂ” browses products, makes purchases, views transactions, uses referrals, downloads invoices
- **Vendor** вЂ” manages product listings, completes KYC on registration, views transactions and invoices

The REST API is not yet built. Flutter and backend are developed in parallel against a shared API contract defined in this document.

---

## 2. Architecture Approach

**Feature-First Clean Architecture** вЂ” single Flutter project, organized into `core/` (shared infrastructure) and `features/` (product features). Clean Architecture layers within each feature. Features never import each other; all shared services live in `core/`.

**Rejected alternatives:**
- Federated packages (Melos) вЂ” premature for this team size and early API discovery phase
- DDD-flavored вЂ” unnecessary ceremony for CRUD-heavy marketplace features

## 6. Core Module Detail

### 6.1 API Client (`core/api/`)

Dio singleton registered as `@singleton` in DI. BaseOptions set from `FlavorConfig.instance.apiBaseUrl`.

**Interceptor chain (in order):**

1. `LoggingInterceptor` вЂ” logs request method, URL, headers, response body. Active only when `FlavorConfig.enableLogging == true`. Uses `pretty_dio_logger`.
2. `AuthInterceptor` вЂ” reads JWT access token from `SecureStorageService`, injects `Authorization: Bearer <token>` header on every request.
3. `ErrorInterceptor` вЂ” maps `DioException` to typed `ServerException`. On HTTP 401: attempts one token refresh via `POST /auth/refresh`; on success stores new tokens and retries original request; on second 401 clears storage and fires `SessionExpired` to trigger logout redirect.

### 6.2 Error Hierarchy (`core/error/`)

```dart
sealed class Failure extends Equatable {
  final String message;
  const Failure(this.message);
}

class NetworkFailure    extends Failure { ... }  // no internet
class ServerFailure     extends Failure {         // 4xx/5xx
  final int? statusCode;
  final String? serverMessage;
}
class AuthFailure       extends Failure { ... }  // 401/403
class CacheFailure      extends Failure { ... }  // storage read/write fail
class ValidationFailure extends Failure {         // 422
  final Map<String, String> fieldErrors;
}
class UnknownFailure    extends Failure { ... }
```

No exceptions cross layer boundaries. DataSources throw `Exception` subtypes; Repository implementations catch them and return `Left(Failure)`.

`error_handler.dart` provides a static `mapExceptionToFailure(Exception e) в†’ Failure` helper used by all repository implementations to avoid duplicating the exceptionв†’failure mapping logic.

### 6.3 Base State Pattern (`core/bloc/`)

```dart
sealed class AppState<T> extends Equatable {
  const AppState();
}

class InitialState<T>  extends AppState<T> { ... }
class LoadingState<T>  extends AppState<T> { ... }
class SuccessState<T>  extends AppState<T> { final T data; ... }
class ErrorState<T>    extends AppState<T> { final Failure failure; ... }
```

All BLoCs and Cubits emit these base variants. Feature-specific states extend or alias them as needed.

### 6.4 Router (`core/router/`)

GoRouter with a single `redirect` function (`route_guard.dart`) that enforces:

1. **Auth guard** вЂ” unauthenticated requests to protected routes в†’ `/login`
2. **Role guard** вЂ” authenticated buyer on `/vendor/*` в†’ `/buyer/home`; vendor on `/buyer/*` в†’ `/vendor/home`
3. **Already-logged-in guard** вЂ” authenticated user landing on `/login` or `/register` в†’ their home

**Route tree:**

```
/splash
/login
/register
/register/kyc           (vendor only, post-register step)
/forgot-password

/buyer
  /buyer/home
  /buyer/transactions
  /buyer/transactions/:id
  /buyer/refer
  /buyer/invoices
  /buyer/invoices/:id

/vendor
  /vendor/home
  /vendor/products
  /vendor/products/create
  /vendor/products/:id/edit
  /vendor/transactions
  /vendor/transactions/:id
  /vendor/invoices
  /vendor/invoices/:id
```

### 6.5 Dependency Injection (`core/di/`)

GetIt service locator + Injectable code generation.

| Lifecycle       | Annotation      | Used for                                    |
|-----------------|-----------------|---------------------------------------------|
| Singleton       | `@singleton`    | Dio, SecureStorageService, FlavorConfig     |
| Lazy Singleton  | `@lazySingleton`| AppRouter, ConnectivityService              |
| Factory         | `@injectable`   | All BLoCs, UseCases, Repositories           |

Run `flutter pub run build_runner build --delete-conflicting-outputs` after adding any new `@injectable` class to regenerate `injection.config.dart`.

### 6.6 Storage (`core/storage/`)

- `SecureStorageService` вЂ” wraps `flutter_secure_storage`. Stores: `access_token`, `refresh_token`, `user_id`. Never stored in plain SharedPreferences.
- `PreferencesService` вЂ” wraps `shared_preferences`. Stores: theme mode, locale, `onboarding_seen` flag.

### 6.7 Theme (`core/theme/`)

Material 3. `app_colors.dart` defines a token-based color palette (no raw hex values in widgets). `app_dimensions.dart` centralises spacing, border radii, icon sizes. Light and dark `ThemeData` both built from the same token set.

### 6.8 Connectivity (`core/network/`)

`ConnectivityService` exposes `Stream<bool> get isConnected`. Root `app.dart` listens and shows a persistent `AppSnackbar` offline banner when `false`.

## 7. State Management

### AppBlocObserver

Registered globally in `bootstrap.dart` via `Bloc.observer = AppBlocObserver()`.

- **Dev/Staging:** logs every state change and event via `logger`
- **Prod:** routes `onError` to analytics service; suppresses verbose change logs

### BLoC data flow (login example)

```
Screen dispatches LoginRequested(email, password)
  в†’ AuthBloc.on<LoginRequested>
    в†’ LoginUseCase(LoginParams(...))
      в†’ AuthRepository.login()
        в†’ AuthRemoteDataSource.login()  [Dio POST /auth/login]
          в†’ returns AuthResponseModel
        в†’ saves JWT to SecureStorageService
        в†’ maps UserModel в†’ UserEntity
      в†’ returns Either<Failure, UserEntity>
    в†’ fold:
        Left(failure)  в†’ emit AuthError(failure)
        Right(user)    в†’ emit AuthSuccess(user)
          в†’ GoRouter redirect based on user.role
```

## 8. Network & API Contract

### API response shape (agreed contract for backend team)

```json
// Success (2xx)
{ "success": true, "data": { ... }, "message": "..." }

// Validation error (422)
{ "success": false, "message": "Validation failed",
  "errors": { "field": "message" } }

// Error (4xx / 5xx)
{ "success": false, "message": "Human-readable error" }
```

### Token refresh flow

```
Request в†’ 401 received
  в†’ POST /auth/refresh { refreshToken }
  в†’ 200: store new tokens, retry original request with new access token
  в†’ 401: clear SecureStorage, emit SessionExpired, navigate to /login
```

## 9. KYC Flow (Vendor)

KYC is triggered immediately after vendor registration, before accessing the vendor shell.

**Steps:**
1. Personal details form (name, date of birth, address)
2. Document upload вЂ” `image_picker` opens camera or gallery; `permission_handler` requests camera + storage permissions; file uploaded as multipart via `POST /kyc/document`
3. Selfie capture вЂ” camera only; uploaded via `POST /kyc/selfie`
4. Submitted state вЂ” screen shows "Under Review" UI; app polls `GET /kyc/status` every 10 seconds until status is `approved` or `rejected`. Push notifications are a future enhancement once a notification service is chosen.

**Route:** `/register/kyc` вЂ” only accessible immediately after registration with `role == vendor`. GoRouter guard prevents direct navigation.

## 11. Testing Strategy

### Unit tests (`test/unit/`)

- Use cases: mock repository, verify `Either` result
- Repository impls: mock datasources, verify modelв†’entity mapping and failure conversion
- No Flutter dependency; run in milliseconds

### BLoC tests (`test/bloc/`)

- Use `bloc_test` package with `blocTest<BLoC, State>()`
- Verify state sequences: e.g., `[LoadingState, SuccessState(user)]`
- Mock use cases with `mocktail`
- Target: 100% BLoC state transition coverage

### Widget tests (`test/widget/`)

- Pump screens with pre-configured BLoC states using `MockBloc`
- Verify correct UI renders for `LoadingState`, `SuccessState`, `ErrorState`
- Verify role-based widget visibility (vendor-only widgets hidden from buyers)

### Integration tests (`integration_test/`)

- Full login в†’ home navigation flow per flavor
- Vendor KYC upload flow
- Run on real device or emulator: `flutter test integration_test/ --flavor dev`

## 13. SOLID Principles Applied

| Principle | Application |
|-----------|-------------|
| **S** вЂ” Single Responsibility | Each use case does exactly one thing. DataSources only fetch. Repositories only transform. |
| **O** вЂ” Open/Closed | New features add new feature folders; core is not modified. New failures extend `Failure` sealed class. |
| **L** вЂ” Liskov Substitution | `AuthRepositoryImpl` is fully substitutable for `AuthRepository` abstract class. |
| **I** вЂ” Interface Segregation | `AuthRepository` only declares auth methods. `ProductRepository` only declares product methods. |
| **D** вЂ” Dependency Inversion | BLoCs depend on use case abstractions. Use cases depend on repository abstractions. Concrete implementations injected by DI. |
