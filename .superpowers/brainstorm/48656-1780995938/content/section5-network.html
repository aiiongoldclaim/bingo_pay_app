Section 5 — Network & Error Handling

Dio with 3 interceptors · Either<Failure, T> across all layers · typed failure hierarchy

Request lifecycle

1. LoggingInterceptor — logs method, URL, headers in dev only
↓ onRequest
2. AuthInterceptor — reads JWT from SecureStorage, injects Authorization: Bearer <token>
↓ onResponse / onError
3. ErrorInterceptor — maps HTTP status → typed Exception; on 401 attempts token refresh once, then triggers logout
↓
DataSource — wraps Dio call in try/catch, throws typed ServerException
↓
Repository Impl — catches exceptions, returns Left(Failure) or Right(Entity)
↓
UseCase → BLoC — result.fold(emitError, emitSuccess)

Token refresh (401 flow)

auth_interceptor.dart (on 401)
on 401 received: → call POST /auth/refresh with refreshToken → if success: store new tokens, retry original request → if refresh also 401: clear storage, fire SessionExpired event → router redirects to /login

Failure hierarchy

core/error/failures.dart
sealed class Failure extends Equatable { final String message; const Failure(this.message); } class NetworkFailure extends Failure { ... } class ServerFailure extends Failure { final int? statusCode; final String? serverMessage; } class AuthFailure extends Failure { ... } class CacheFailure extends Failure { ... } class ValidationFailure extends Failure { final Map<String, String> fieldErrors; } class UnknownFailure extends Failure { ... }

API response contract

Expected JSON shape from your REST API
// Success { "success": true, "data": { ... }, "message": "Login successful" } // Validation error (422) { "success": false, "message": "Validation failed", "errors": { "email": "Already in use", "password": "Too short" } } // Server error (4xx / 5xx) { "success": false, "message": "Something went wrong" }
Contract-first benefit: Since your REST API isn't built yet, defining this response shape now means your Flutter data layer and backend can be built in parallel with zero integration surprises.
