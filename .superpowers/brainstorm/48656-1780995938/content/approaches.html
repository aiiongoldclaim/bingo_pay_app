Architecture Approach Options

Choose the structural foundation for Bingo Pay

A

Feature-First Monolith Recommended

Single Flutter project. lib/ split into core/ and features/. Clean Architecture within each feature. Feature isolation is logical (enforced by convention + barrel exports).

Pros

  • Standard, well-understood structure
  • Fastest to bootstrap
  • Easy cross-feature navigation
  • Single pubspec.yaml
  • Can graduate to packages later

Cons

  • Isolation is by convention only
  • Large teams can bleed feature boundaries
Best for: greenfield project, API not yet built, team finding domain boundaries
B

Federated Package Architecture (Melos)

Each feature is a separate Dart package under packages/features/. Core is a shared package. Melos manages the monorepo. True compile-time isolation between features.

Pros

  • True Dart-level feature isolation
  • Independent package versioning
  • Compile errors on cross-feature leaks
  • Scales to 20+ features / large teams

Cons

  • High setup + maintenance overhead
  • Refactoring across packages is painful
  • Premature for early-stage API discovery
  • pubspec.yaml per feature, path dependencies
Best for: 5+ engineers, stable domain, features owned by separate teams
C

DDD-Flavored Feature-First

Single package like Option A, but with richer Domain-Driven Design concepts: Value Objects (Money, EmailAddress), Aggregates (Order, Vendor), and formal Bounded Contexts per feature.

Pros

  • Excellent for complex payment domain logic
  • Type-safe domain primitives (Money, TaxId)
  • Business rules live in domain layer

Cons

  • More boilerplate than needed for CRUD
  • Steeper learning curve for new devs
  • Most marketplace features are CRUD-heavy
Best for: fintech apps with complex business rules; overkill for standard marketplace CRUD
